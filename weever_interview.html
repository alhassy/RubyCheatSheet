Product Distribution with ‘Robots’

Abstract

Let's write some Ruby code to model distribution companies —such as Amazon.

We'll suppose that distribution is automated via robots that can advance only one step in any cardinal direction. As such, we'll make a class for robots that models their behaviour then a class for the companies —which, in turn, consist of a number of robots.

This is a solution to one of WeeverApps’ interview problems.

The full code is at the very end, and also here.

1 Problem Statement

From weeverapps…

Robbie the robot and his pals deliver presents to houses. Build a program that runs a simulation given a number of robots and a string of their orders, keeps track of the state of the robots, and the presents they’ve delivered.

The robots move in the 4 cardinal directions. They each take turns. On a robot’s turn they will read one instruction from the moves sequence, move in the indicated direction, and attempt to deliver their present.

A robot may deliver a present if, and only if, there are no other robots on the space in which they are ending their turn.

For example, if there are 3 robots: robbie, jane, bob, and a move sequence of ^^VV<> we should get the following moves:

robbie ^
jane ^
bob V
robbie V
jane <
bob >

A present is delivered when a robot enters a space in the world.

For readability, we will aim to produce user-friendly output along with the map of the world visited by our robots. For instance, using inputs "jasim, kathy, bobert" and ">>>^>v", below one can see the map grow horizontally or vertically as robots perform deliveries. 

1. jasim   goes right	--moving from (0, 0) to (1, 0)
⟨0⟩ 1

2. kathy   goes right	--moving from (0, 0) to (1, 0)
⟨0⟩ 2

3. bobert  goes right	--moving from (0, 0) to (1, 0)
⟨0⟩ 3

4. jasim   goes up      --moving from (1, 0) to (1, 1)
 0  1
⟨0⟩ 3

5. kathy   goes right	--moving from (1, 0) to (2, 0)
 0  1  0
⟨0⟩ 3  1

6. bobert  goes down    --moving from (1, 0) to (1, -1)
 0  1  0
⟨0⟩ 3  1
 0  1  0

The numbers denote how many deliveries were made to a particular location; whereas the singular number enclosed in angle brackets, ⟨⋯⟩, is a visual indication of the origin —location (0,0). The benefit of this approach is that one can quickly glean a number of useful statistics.

Going forward , we will display results of examples in these boring square boxes.

2 What's a Robot?

It is an entity that deliviers presents to homes, and as such, it has a name —which should be unique— as well as a current location. For fun, let's also also keep track of it's deliveries.

A robot is responsible for its features, but anyone may ask where it is, its name, and the deliveries it has performed. 

# Robots have two other useful features that one may query.
attr_reader   :name       # Name of the robot
attr_accessor :x, :y      # Current location
attr_reader   :deliveries # Non-empty list of locations it has been to.
                          # A “location” is a hash of two
                          # integers :x, :y.

Robots begin their day at the company wharehouse; i.e., the origin (0,0). 

# A robot is brought into existence needing only a name.
def initialize name
  @name         = name
  @x            = 0
  @y            = 0
  @deliveries = [ {x: @x, y: @y} ]
  end

For our aimed output —as discussed in §1— we need to pretty print robot locations. 

# Its current location, a pair of integers
def location
    "(#{@x}, #{@y})"
end

Finally, we're at the heart of the matter: Robot movement. We pattern match on the shape of the direction symbol and act accordingly, if possible; otherwise, we raise a helpful error message. 

def move direction
  case direction
  in :up
    @y += 1
  in :down
    @y -= 1
  in :left
    @x -= 1
  in :right
    @x += 1
  in oops
    raise "#{@name} sees a malformed direction: #{oops}!"\
         "\nCardinal directions are the symbols :up, :down, :left, :right."
  end
  @deliveries << {x: @x, y: @y}
end

So much for robots.

Before moving on, let's have a super simple sanity check. That is, let's actually test out this class. 

robbie = Robot.new(:robbie)
robbie.move :up
robbie.move :right
robbie.move :left
p robbie.location, robbie.deliveries

"(0, 1)"
[{:x=>0, :y=>0}, {:x=>0, :y=>1}, {:x=>1, :y=>1}, {:x=>0, :y=>1}]

3 What's a distribution company?

A distribution company has a number of robots along with a mutable list of deliveries to be performed. Even though robots keep track of their deliveries —i.e., we have a robot-to-location mapping—, it is useful to have the converse view as well: Given a location, which robots have gone there? This is how we get the number of deliveries shown in the introduction, §1. 

attr_accessor :universe   # A mapping of locations-to-robots
attr_accessor :directions # List of directions
attr_accessor :robots     # List of directions

Users need not be burdened with the need to write [:right, :right, :right, :up, :right, :down] when ">>>^>v" achieves the same goal and is nicer for the user. Likewise for robot names. There is a clear mapping between the two representations, as shown below. 

# In Robot#move we were verbose, using pattern matching and error checking.
# This time, let's forgoe the checks and be terse.
# That is, a precondition of this method if that
# ‘arg’ is a string consisting of ^,v,V,<,> .
def directions= arg
  @directions = arg.split('').map{ |x| {"^" => :up,
                                        "v" => :down, "V" => :down,
                                        "<" => :left, ">" => :right}[x] }
end

# A precondition of this method is that
# ‘arg’ is a comma separated string of unique names.
def robots= arg
  @robots = arg.split(",").map{ |x| Robot.new(x.strip.to_sym) }
end

Using the above setter methods, a distribution company initialisation is clear and compact. 

def initialize robots = "robbie", directions = ""
  @universe       = Hash.new([]) # Thus far, every location has vistiors [].
  self.robots     = robots
  self.directions = directions

  # Universe dimensions grow with robot movements.
  @X = [0, 0]
  @Y = [0, 0]
  end

We have enough matter to run some tests —no need to keep going if things don't work at this stage! 

weever = DistributionsCompany.new "jasim, kathy, bobert" , ">>>^>v"
p weever.robots.map(&:name), weever.directions

weever.directions = "Vv<>"
weever.robots = "robbie, jane, roy"
p weever.robots.map(&:name), weever.directions

[:jasim, :kathy, :bobert]
[:right, :right, :right, :up, :right, :down]
[:robbie, :jane, :roy]
[:down, :down, :left, :right]

Excellent!

Universe dimensions?

In order to draw the map of locations visited by our robots, we need to know the dimensions of the universe. Instead of having four variables — @@X_min, @@X_max, @@Y_min, @@Y_max —, we group the min-max of each dimension with an eye toward making use of Enumerable#minmax.

4 How big is our ‘universe’

Since @X, @Y keep track of the universe dimensions, we can now show the map by printing the number of robots that have visited a location —and emphasising the origin. 

def show
  puts ""
  for row in @Y[1].downto @Y[0] do
    for col in @X[0].upto @X[1] do
      format = if row == 0 and col == 0 then "⟨%s⟩" else " %s " end
      print format % @universe[{x: col, y: row}].length
    end
    puts ""
  end
end

The universe does not yet grow, let's change that! When a company ships its products, the robots will move and, in turn, we will know which parts of the city, landscape, our robots have traversed. 

 1: # This method carries out a number of directions.
 2: #
 3: # step: The number of directional steps to perform; all by default.
 4: # withmap: Flag to indicate whether a map should be drawn at each stage.
 5: #          For brevity, this requires that step be non-nil.
 6: def ship (step: nil, withmap: false)
 7:   # Essentially zipping with a cyclic list
 8:   l = @robots.length
 9:   (@directions || []).each_with_index do |d, i|
10: 
11:     # Move the current robot
12:     r            = @robots[i.modulo(l)]
13:     old_location = r.location
14:     r.move(d)
15: 
16:     # Update the universe dimensions
17:     @universe[{x: r.x, y: r.y}] += [r.name]
18:     @Y = (@Y + [r.y]).minmax
19:     @X = (@X + [r.x]).minmax
20: 
21:     # Print the map incrementally, if requested
22:     unless not step
23:       break unless i < step
24:       print "\n#{1 + i}. #{r.name.to_s.ljust(7)} goes #{d.to_s.ljust(5)}"\
25:             "\t--moving from #{old_location} to #{r.location}"
26:       show if withmap
27:     end
28:   end
29: end

This is, perhaps, the most crucial method. Let's briefly discuss its design.

Line 8
We're essentially ‘zipping’ a pair of lists, but we want to treat the robots list as if it were an infinite cyclic list. We simulate this behaviour by controlling access to it: When an index is out of bounds, we wrap it around to the beginning —as shown in Line 12.
Line 9
The list of directions may have been fully consumed previous shipment, and so be nil. When that happens, the loop happens on the empty list —and so ‘nothing happens’, unexceptionally.
Line 13
We keep the old location for pretty printing only.
Line 16

If we had used the 4 variable approach, we'd have the following ghastly code! 

@Y_max = [@Y_max, y].max
@X_max = [@X_max, x].max
@Y_min = [@Y_min, y].min
@X_min = [@X_min, x].min
Line 21
Boring code to get the job done: Unless we've shown the requested number of step-s, we print the current robot's movement and, if requested, we draw the map of the universe.

Due to time constraints, a practical feature has been omitted: After shipping, we should discard the directions actually followed (and possibly reset the locations of the robots to the origin); otherwise future calls to ship would result in duplicate deliveries to the same places.

6 Where are my bots?

We can extract out some useful aspects of the ship method into top-level functions… 

# Pretty prints a listing of the robots’ positions
def positions
  for r in @robots do
    puts "#{r.name.to_s.ljust(7)} is at #{r.location}"
  end
end

# How many houses have at least n presents?
def presents n
  @universe.values.map(&:length).filter {|x| x >= n}.length
end
‘presents’ test 
weever = DistributionsCompany.new "jasim, kathy, bobert" , ">>>^>v"
weever.ship
weever.show
puts "\nNumber of locations with ≥ 1 present is #{weever.presents 1}"

0  1  0
⟨0⟩ 3  1
 0  1  0

Number of locations with ≥ 1 present is 4

# How many deliveries were performed?
def total_deliveries
  @universe.values.map(&:length).sum
end
‘total_deliveries’ test 
weever = DistributionsCompany.new "jasim, kathy, bobert" , ">>>^>v"
weever.ship
weever.show
puts "\nThe total number of deliveries is #{weever.total_deliveries}."

0  1  0
⟨0⟩ 3  1
 0  1  0

The total number of deliveries is 6.

The above methods could have been, for efficiency, accumulated in the ship method —rather than incurring the linear cost of accessing @robots and @universe multiple times.
